为什么读 omiGlass？

虽然 MCU 是 Xtensa 不是 Cortex-M，但技术栈高度重叠：I2S PDM 音频采集、Opus 编码、BLE 数据传输、功耗管理、OTA 升级的思路可直接迁移到 Cortex-M 项目。

音频管线

PDM Mic → I2S → Opus 编码 → BLE 传输，与 STM32 SAI + LC3 的架构如出一辙

功耗管理

CPU 降频、Light Sleep、Deep Sleep，与 STM32 Stop/Standby 模式设计思路相同

OTA 升级

双分区 Flash 布局 + 安全回滚，与 STM32 双 Bank 和 nRF5340 DFU 对比学习

②自定义分区表：ESP32 默认分区不支持 OTA，必须用自定义 CSV 文件定义双 App 分区。STM32 通过 linker script (.ld) 定义内存布局，ESP32 通过分区表。
③qio_opi：XIAO ESP32-S3 使用 OPI PSRAM（8 线），比传统 SPI PSRAM 带宽翻倍。音频缓冲区分配在 PSRAM 中。

⑨NimBLE：比 BLE 4.2 协议栈节省 ~50% RAM。智能眼镜内存紧张，NimBLE 是必选。

// 设备生命周期状态
typedef enum {
    DEVICE_BOOTING,      // ① 启动中：LED 快闪，初始化各子系统
    DEVICE_ACTIVE,       // ② 正常工作：BLE + 音频 + 相机
    DEVICE_POWER_SAVE,   // ③ 省电模式：CPU 降到 40MHz
    DEVICE_LOW_BATTERY,  // ④ 低电：最小化操作
    DEVICE_POWERING_OFF, // ⑤ 关机中：LED 闪烁序列
    DEVICE_SLEEP         // ⑥ 深度睡眠：仅按钮唤醒
} device_state_t;

// 功耗状态（与设备状态独立管理）
typedef enum {
    POWER_STATE_ACTIVE,      // 正常操作
    POWER_STATE_POWER_SAVE,  // 降频运行
    POWER_STATE_LOW_BATTERY, // 最小操作
    POWER_STATE_SLEEP        // Deep Sleep
} power_state_t;

// 按钮状态（消抖状态机）
typedef enum {
    BUTTON_IDLE,        // 空闲
    BUTTON_PRESSED,     // ⑦ 刚按下（等待判断长短按）
    BUTTON_LONG_PRESS,  // ⑧ 持续 2 秒 = 关机
    BUTTON_RELEASED     // 释放
} button_state_t;

// LED 显示模式
typedef enum {
    LED_OFF,
    LED_ON,
    LED_BOOT_SEQUENCE,     // 启动：5 次快闪 1.5s
    LED_NORMAL_OPERATION,  // 运行：连接=常亮，未连接=慢闪
    LED_LOW_BATTERY,       // 低电：1s 间隔闪烁
    LED_PHOTO_CAPTURE,     // 拍照：100ms 闪
    LED_POWER_OFF_SEQUENCE,// 关机：2 次快闪 0.8s
    LED_SLEEP_MODE         // 睡眠：5s 超慢闪
} led_status_t;
⑦⑧按钮状态机：短按（>50ms）= 唤醒/活跃，长按（≥2s）= 关机。消抖 50ms 防误触。这是嵌入式按钮处理的经典模式。
→状态机解耦：device_state_t 管理设备生命周期，led_status_t 管理显示行为，两者独立。设备状态转换不直接操作 GPIO，而是通过 LED 模式间接表达。

bool lightSleepEnabled = true;       // Light Sleep 使能

// GPIO 中断服务函数（RAM 中执行）
void IRAM_ATTR buttonISR() {   // ② IRAM_ATTR 确保在 Flash 访问时仍可触发
    buttonPressed = true;       // 仅设标志，不在 ISR 中做复杂操作
}

// 按钮处理（每次 loop 调用）
void handleButton() {
    unsigned long now = millis();
    static unsigned long lastDebounceTime = 0;
    static bool buttonDown = false;
    static bool longPressTriggered = false;

    bool currentButtonState = !digitalRead(POWER_BUTTON_PIN); // ③ 活跃低

    if (currentButtonState && !buttonDown) {
        if (now - lastDebounceTime < 50) return;  // ④ 50ms 消抖
        buttonPressTime = now;
        buttonDown = true;
        longPressTriggered = false;
    } else if (currentButtonState && buttonDown && !longPressTriggered) {
        if (now - buttonPressTime >= 2000) {  // ⑤ 长按 2s = 关机
            longPressTriggered = true;
            ledMode = LED_POWER_OFF_SEQUENCE;
        }
    } else if (!currentButtonState && buttonDown) {
        if (now - lastDebounceTime < 50) return;
        buttonDown = false;
        if (!longPressTriggered && (now - buttonPressTime) >= 50) {
            lastActivity = now;        // ⑥ 短按 = 注册活动
            if (powerSaveMode) exitPowerSave();
        }
    }
    buttonPressed = false;
}
①②ISR 最小化原则：中断只设 volatile 标志，复杂逻辑在主循环处理。IRAM_ATTR 确保 ISR 代码在 RAM 中执行，不会被 Flash 访问阻塞。与 STM32 的 EXTI 中断设计相同。
④消抖：50ms 内的连续触发被忽略。嵌入式按钮消抖的标配做法 — 也可以用硬件 RC 滤波。
⑤⑥长短按区分：持续 2s → 关机序列；释放时 >50ms → 短按（唤醒/退出省电）。用一个 static bool 防止长按重复触发。

app.cpp:128-187 — LED 控制逻辑
void updateLED() {
    unsigned long now = millis();
    static unsigned long bootStartTime = 0;
    static unsigned long powerOffStartTime = 0;

    switch (ledMode) {
    case LED_BOOT_SEQUENCE:       // ① 启动：5 次快闪，1.5s 完成
        if (bootStartTime == 0) bootStartTime = now;
        if (now - bootStartTime < 1500) {
            int blinkPhase = ((now - bootStartTime) / 150) % 2;
            digitalWrite(STATUS_LED_PIN, !blinkPhase);  // ② 反逻辑：HIGH=OFF
        } else {
            digitalWrite(STATUS_LED_PIN, HIGH); // OFF
            ledMode = LED_NORMAL_OPERATION;
        }
        break;

    case LED_POWER_OFF_SEQUENCE:  // ③ 关机：2 次快闪，0.8s 后进入 Deep Sleep
        if (powerOffStartTime == 0) powerOffStartTime = now;
        if (now - powerOffStartTime < 800) {
            int blinkPhase = ((now - powerOffStartTime) / 200) % 2;
            digitalWrite(STATUS_LED_PIN, !blinkPhase);
        } else {
            digitalWrite(STATUS_LED_PIN, HIGH);
            delay(100);
            shutdownDevice();       // 关机
        }
        break;

    case LED_NORMAL_OPERATION:    // ④ 正常运行
    default:
        if (connected) {
            digitalWrite(STATUS_LED_PIN, LOW);   // ⑤ 已连接：常亮
        } else {
            int blinkPhase = (now / 1000) % 2;
            digitalWrite(STATUS_LED_PIN, blinkPhase ? HIGH : LOW); // ⑥ 未连接：1s 慢闪
        }
        break;
    }
}

void blinkLED(int count, int delayMs) {  // ⑦ 通用闪烁函数
    for (int i = 0; i < count; i++) {
        digitalWrite(STATUS_LED_PIN, HIGH);  // OFF
        delay(delayMs);
        digitalWrite(STATUS_LED_PIN, LOW);   // ON
        delay(delayMs);
    }
}
②GPIO21 反逻辑：XIAO ESP32-S3 的用户 LED 是低电平点亮（HIGH=灭，LOW=亮）。这在嵌入式中很常见 — NPN 驱动 LED 就是这个逻辑。代码中统一用 ! 取反。
⑤⑥连接状态可视化：用户只需看 LED 就知道设备状态 — 常亮=已连接手机，慢闪=等待连接。这是没有屏幕的嵌入式设备的基本交互设计。

LED 模式速查

  • 5 次快闪 1.5s → 启动中
  • 常亮 → 已连接手机
  • 1s 慢闪 → 等待连接
  • 2 次快闪 0.8s → 正在关机
  • 5s 超慢闪 → 深度睡眠

设计模式：枚举驱动显示

led_status_t 枚举将设备状态与 LED 行为解耦。设备状态转换时只需修改 ledMode 变量，updateLED() 在每次 loop 中根据当前模式渲染 LED。类似 MVC 模式 — 状态是 Model，LED 是 View。

mic.cpp — I2S PDM 配置与数据采集

mic.cpp:15-87 — 初始化与 I2S 配置
bool mic_start() {
    // ① PSRAM 分配（优先）+ 常规 RAM 兜底
    i2s_read_buffer = (int16_t *) ps_malloc(MIC_BUFFER_SAMPLES * sizeof(int16_t));
    if (i2s_read_buffer == nullptr) {
        i2s_read_buffer = (int16_t *) malloc(MIC_BUFFER_SAMPLES * sizeof(int16_t));
        // 分配失败则返回 false
    }

    // ② I2S PDM 配置
    i2s_config_t i2s_config = {
        .mode = (i2s_mode_t)(I2S_MODE_MASTER | I2S_MODE_RX | I2S_MODE_PDM),  // ③ PDM 接收
        .sample_rate = MIC_SAMPLE_RATE,           // 16kHz
        .bits_per_sample = I2S_BITS_PER_SAMPLE_16BIT,  // 16-bit
        .channel_format = I2S_CHANNEL_FMT_ONLY_LEFT,   // ④ 单声道
        .communication_format = I2S_COMM_FORMAT_STAND_I2S,
        .intr_alloc_flags = ESP_INTR_FLAG_LEVEL1,
        .dma_buf_count = 8,     // ⑤ 8 个 DMA 缓冲区
        .dma_buf_len = 256,     // 每个缓冲 256 字节
        .use_apll = false,      // 不用 APLL（精度够用）
    };

    // ⑥ 引脚配置 — XIAO ESP32S3 Sense 内置 PDM 麦克风
    i2s_pin_config_t pin_config = {
        .bck_io_num = I2S_PIN_NO_CHANGE,   // PDM 不需要 BCK
        .ws_io_num = MIC_CLK_PIN,           // GPIO42 = PDM CLK
        .data_out_num = I2S_PIN_NO_CHANGE,  // 只接收不发送
        .data_in_num = MIC_DATA_PIN,        // GPIO41 = PDM DATA
    };

    i2s_driver_install(I2S_PORT, &i2s_config, 0, NULL);  // ⑦ 安装驱动
    i2s_set_pin(I2S_PORT, &pin_config);                    // 配置引脚
    i2s_zero_dma_buffer(I2S_PORT);                         // 清空 DMA 缓冲
}
①PSRAM 优先策略：ESP32-S3 有 8MB PSRAM（外部 SRAM），适合存放大缓冲区。ps_malloc() 尝试 PSRAM，失败则用内部 RAM。对比 STM32 的 SRAM 紧张场景，ESP32 的 PSRAM 是巨大优势。
③PDM 模式：MEMS 数字麦克风的输出格式。与 I2S 的区别：PDM 只需 2 根线（CLK+DATA），I2S 需要 3-4 根线（BCK+WS+DATA+MCLK）。nRF5340 有硬件 PDM 外设，STM32 需要用 SAI 的 PDM 模式。
⑤8×256 DMA 缓冲：8 个缓冲区 × 256 字节 = 2KB DMA 缓冲。16kHz × 2 字节 = 32KB/s 数据率，每个缓冲可存 128 个样本 = 8ms。8 个缓冲 = 64ms 延迟容忍。与 libDaisy 的 Ping-Pong DMA 类似，但 ESP32 I2S 驱动内部管理 DMA 链。
mic.cpp:114-144 — 数据处理与增益
void mic_process() {
    if (!mic_running || i2s_read_buffer == nullptr) return;

    size_t bytes_read = 0;
    // ① 非阻塞读取，20ms 超时
    esp_err_t err = i2s_read(I2S_PORT, i2s_read_buffer,
        MIC_BUFFER_SAMPLES * sizeof(int16_t), &bytes_read, pdMS_TO_TICKS(20));

    if (err == ESP_OK && bytes_read > 0) {
        size_t samples_read = bytes_read / sizeof(int16_t);

        // ② 增益处理：int32 放大 + 16-bit 钳位
        if (MIC_GAIN != 1) {
            for (size_t i = 0; i < samples_read; i++) {
                int32_t sample = (int32_t)i2s_read_buffer[i] * MIC_GAIN;
                if (sample > 32767)  sample = 32767;   // ③ 正向钳位
                if (sample < -32768) sample = -32768;  // 负向钳位
                i2s_read_buffer[i] = (int16_t)sample;
            }
        }
        if (audio_callback != nullptr)
            audio_callback(i2s_read_buffer, samples_read);  // ④ 回调通知
    }
}
②③增益 ×2：PDM 麦克风输出信号较小，需要软件放大。先转 int32 避免溢出，再钳位到 16-bit 范围。这是嵌入式音频的常规操作 — 类似模拟前端的 PGA（可编程增益放大器），但在数字域实现。

①PSRAM 缓冲策略：16KB 环形缓冲区存 8000 个样本 = 500ms @ 16kHz。即使编码器短暂阻塞，也能缓冲 500ms 数据。内部 RAM 只有 512KB，音频缓冲放 PSRAM 释放宝贵的内部 RAM。
③VOIP 模式：Opus 有三种模式 — VOIP（语音优化）、AUDIO（音乐优化）、RESTRICTED_LOWDELAY。智能眼镜传输语音，VOIP 模式在 32kbps 下提供最佳语音质量。

// ① 环形缓冲区写入（被 mic 回调调用）
int opus_receive_pcm(int16_t *data, size_t samples) {
    for (size_t i = 0; i < samples; i++) {
        size_t next_write = (ring_write_pos + 1) % AUDIO_RING_BUFFER_SAMPLES;
        if (next_write == ring_read_pos) {
            ring_read_pos = (ring_read_pos + 1) % AUDIO_RING_BUFFER_SAMPLES; // ② 丢弃最旧
        }
        pcm_ring_buffer[ring_write_pos] = data[i];
        ring_write_pos = next_write;
    }
    return 0;
}

// 缓冲区可用数据量
static size_t ring_buffer_available() {
    if (ring_write_pos >= ring_read_pos)
        return ring_write_pos - ring_read_pos;
    else
        return AUDIO_RING_BUFFER_SAMPLES - ring_read_pos + ring_write_pos;
}

// ③ 编码处理（主循环调用）
void opus_process() {
    while (ring_buffer_available() >= OPUS_FRAME_SAMPLES) {  // ④ 凑够 320 样本
        // 从环形缓冲区取一帧
        for (size_t i = 0; i < OPUS_FRAME_SAMPLES; i++) {
            opus_input_buffer[i] = pcm_ring_buffer[ring_read_pos];
            ring_read_pos = (ring_read_pos + 1) % AUDIO_RING_BUFFER_SAMPLES;
        }
        // 编码
        int encoded_bytes = opus_encode_frame(opus_input_buffer, OPUS_FRAME_SAMPLES);
        if (encoded_bytes > 0 && encoded_callback != nullptr)
            encoded_callback(opus_output_buffer, encoded_bytes);  // ⑤ 回调通知
    }
}
②丢弃策略：环形缓冲区满时丢弃最旧数据（读指针前进），保证最新音频不被延迟。音频系统宁可丢旧数据也不能阻塞 — 这是实时音频的基本原则。
④20ms 帧粒度：320 samples @ 16kHz = 20ms。Opus 编码器要求固定帧长。每 20ms 产生一个编码包，BLE 每秒发 50 个包。

为什么 16kHz / 32kbps / 20ms？

  • • 16kHz：语音频宽 8kHz 足够，16kHz 留余量
  • • 32kbps：Opus 在 32kbps 语音质量接近原始 PCM
  • • 20ms：BLE 连接间隔 25-50ms，20ms 帧正好适配
  • • 320 samples/帧：16000 × 0.02 = 320

编解码器对比

  • • Opus：6-510kbps，延迟 2.5-60ms，开源，语音+音乐
  • • LC3：BLE LE Audio 专用，同码率质量接近 Opus
  • • SBC：经典蓝牙，170-345kbps，质量最差
  • • μ-law：64kbps，8kHz，最简单（omiGlass 旧版使用）

mulaw.h — 遗留 μ-law 编码（简述）

mulaw.h（44 行）实现了 16-bit PCM → 8-bit μ-law 的压缩编码。这是 omiGlass 早期版本的音频编码方案，后被 Opus 完全替代。μ-law 将 16-bit 压缩到 8-bit（2:1 压缩），码率 128kbps @ 16kHz，比 Opus 的 32kbps 高 4 倍。文件保留作为历史参考，不再在生产代码路径中使用。

void onOpusEncoded(uint8_t *data, size_t len) {
    if (len > OPUS_OUTPUT_MAX_BYTES) return;
    size_t packet_size = len + 2;  // 2 字节长度头
    // 写入 TX 环形缓冲区（length + data）
    audio_tx_buffer[audio_tx_write_pos] = len & 0xFF;
    audio_tx_buffer[(audio_tx_write_pos + 1) % AUDIO_TX_BUFFER_SIZE] = (len >> 8) & 0xFF;
    for (size_t i = 0; i < len; i++)
        audio_tx_buffer[(audio_tx_write_pos + 2 + i) % AUDIO_TX_BUFFER_SIZE] = data[i];
    audio_tx_write_pos = (audio_tx_write_pos + packet_size) % AUDIO_TX_BUFFER_SIZE;
}

// ③ 构建音频包：3 字节头 + payload
void broadcastAudioPacket(uint8_t *data, size_t len) {
    if (!connected || !audioSubscribed) return;
    audio_packet_buffer[0] = audioPacketIndex & 0xFF;        // 帧索引低字节
    audio_packet_buffer[1] = (audioPacketIndex >> 8) & 0xFF; // 帧索引高字节
    audio_packet_buffer[2] = 0;  // ④ 子索引（分片用，当前不用）
    memcpy(audio_packet_buffer + AUDIO_PACKET_HEADER_SIZE, data, len);
    audioDataCharacteristic->setValue(audio_packet_buffer, len + 3);
    audioDataCharacteristic->notify();  // ⑤ BLE Notify 推送
    audioPacketIndex++;
}

// ⑥ TX 环形缓冲区出队 + 发送（主循环调用）
void processAudioTx() {
    if (!connected || !audioSubscribed) return;
    while (audio_tx_read_pos != audio_tx_write_pos) {
        uint16_t len = audio_tx_buffer[audio_tx_read_pos]
                     | (audio_tx_buffer[(audio_tx_read_pos+1) % BUF_SIZE] << 8);
        // 读取数据 → broadcastAudioPacket → 发送
        audio_tx_read_pos = (audio_tx_read_pos + 2 + len) % AUDIO_TX_BUFFER_SIZE;
        broadcastAudioPacket(temp_data, len);
        delay(1);  // ⑦ 1ms 间隔防止 BLE 拥塞
    }
}
③④音频包协议：3 字节头 = 2 字节帧索引（递增，接收端排序）+ 1 字节子索引（大包分片用）。payload 是 Opus 编码数据（通常 40-80 字节）。对比 BLE LE Audio 的 ISO 通道自带时序标记，BLE Notify 需要应用层自己管理序列号。
⑦1ms 限流：BLE Notify 不能连续快速发送，需要间隔。这里用 delay(1) 在每个包之间留 1ms。音频每 20ms 产出一帧编码数据，50Hz 发送频率完全在 BLE 带宽范围内（MTU 517 字节 × 40 连接/秒 = 20KB/s）。

与 nRF5340 Audio 的对比：nRF5340 Audio 使用 BLE LE Audio 的 ISO 同步通道，硬件保证时序、无需应用层序列号。omiGlass 用 BLE Notify + 应用层协议实现，更简单但不保证实时性。这是"标准方案"vs"实用方案"的典型对比。

void enableLightSleep() {
    if (!lightSleepEnabled || !connected || photoDataUploading) return;
    if (millis() - lastActivity < 5000) return;  // 5 秒内有活动不睡

    unsigned long timeUntilNextPhoto = 0;
    if (isCapturingPhotos && captureInterval > 0) {
        // ④ 计算到下次拍照的时间
        unsigned long timeSinceLastPhoto = millis() - lastCaptureTime;
        if (timeSinceLastPhoto < captureInterval)
            timeUntilNextPhoto = captureInterval - timeSinceLastPhoto;
    }

    // 仅当距下次拍照 > 10 秒时才睡
    if (timeUntilNextPhoto > 10000) {
        unsigned long sleepTime = timeUntilNextPhoto - 5000;
        if (sleepTime > 15000) sleepTime = 15000;  // ⑤ 最长睡 15 秒
        esp_sleep_enable_timer_wakeup(sleepTime * 1000);
        esp_light_sleep_start();  // ⑥ 进入 Light Sleep
        lastActivity = millis();  // 唤醒后更新活动时间
    }
}

    // ② ADC 转换：12-bit → 电压
    float adcVoltage = (adcValue / 4095.0f) * 3.3f;

    // ③ 分压器还原真实电池电压
    batteryVoltage = adcVoltage * VOLTAGE_DIVIDER_RATIO;  // ×6.086

    // ④ 负载补偿的百分比计算
    if (batteryVoltage >= BATTERY_MAX_VOLTAGE)
        batteryPercentage = 100;
    else if (batteryVoltage <= BATTERY_MIN_VOLTAGE)
        batteryPercentage = 0;
    else
        batteryPercentage = (int)(((batteryVoltage - BATTERY_MIN_VOLTAGE)
            / (BATTERY_MAX_VOLTAGE - BATTERY_MIN_VOLTAGE)) * 100.0f);

    // ⑤ 平滑处理：变化 > 5% 才更新（防止跳变）
    static int lastBatteryPercentage = batteryPercentage;
    if (abs(batteryPercentage - lastBatteryPercentage) > 5)
        batteryPercentage = lastBatteryPercentage +
            (batteryPercentage > lastBatteryPercentage ? 2 : -2);
}
①⑤抗噪声 + 防跳变：10 次采样平均消除 ADC 噪声，百分比变化限制在 ±2% 防止 UI 跳变。这是电池监测的标准做法 — STM32 也可以用同样的 ADC + 分压器方案。

void ota_handle_command(uint8_t *data, size_t length) {
    uint8_t command = data[0];
    switch (command) {
    case OTA_CMD_SET_WIFI: {  // ① 设置 WiFi 凭据
        // 格式: [cmd, ssid_len, ssid..., pass_len, pass...]
        uint8_t ssidLen = data[1];
        memcpy(wifiSSID, &data[2], ssidLen);
        wifiSSID[ssidLen] = '\0';
        uint8_t passLen = data[2 + ssidLen];
        memcpy(wifiPassword, &data[3 + ssidLen], passLen);
        wifiPassword[passLen] = '\0';
        wifiCredentialsSet = true;
        break;
    }
    case OTA_CMD_SET_URL: {   // ② 设置固件 URL
        // 格式: [cmd, url_len(2B big-endian), url...]
        uint16_t urlLen = (data[1] << 8) | data[2];
        memcpy(firmwareURL, &data[3], urlLen);
        firmwareURL[urlLen] = '\0';
        firmwareURLSet = true;
        break;
    }
    case OTA_CMD_START_OTA: { // ③ 启动 OTA 任务
        if (!wifiCredentialsSet || !firmwareURLSet) {
            ota_notify_status(OTA_STATUS_ERROR); return;
        }
        if (otaTaskRunning) return;  // ④ 防止重复启动
        otaCancelled = false;
        otaTaskRunning = true;
        xTaskCreate(ota_task, "ota_task", 8192, NULL, 5, &otaTaskHandle);  // ⑤ FreeRTOS 任务
        break;
    }
    case OTA_CMD_CANCEL_OTA:  ota_cancel(); break;
    case OTA_CMD_GET_STATUS:  ota_notify_status(otaStatus, otaProgress); break;
    }
}
⑤FreeRTOS 任务：OTA 在独立任务中运行（8KB 栈，优先级 5），不阻塞主循环的音频处理。ESP32 的 Arduino 框架底层就是 FreeRTOS。

static void ota_task(void *parameter) {
    // ① Step 1: WiFi STA 连接
    if (!connect_wifi()) {  // 15 秒超时
        otaTaskRunning = false; vTaskDelete(NULL); return;
    }
    // ② Step 2: 下载并安装固件
    if (!download_and_install_firmware()) {
        WiFi.disconnect(true); otaTaskRunning = false; vTaskDelete(NULL); return;
    }
    // ③ Step 3: 重启
    ota_notify_status(OTA_STATUS_REBOOTING);
    delay(2000);  // 等 BLE 通知发出
    WiFi.disconnect(true); WiFi.mode(WIFI_OFF);
    ESP.restart();  // ④ 重启到新固件
}

static bool download_and_install_firmware() {
    // HTTP/HTTPS 流式下载
    HTTPClient http;
    http.begin(*client, firmwareURL);
    int httpCode = http.GET();  // ⑤ GET 请求
    int contentLength = http.getSize();

    Update.begin(contentLength);  // ⑥ 初始化 Flash 写入

    WiFiClient *stream = http.getStreamPtr();
    uint8_t buffer[1024];  // ⑦ 1KB 块读取
    while (http.connected() && totalRead < contentLength) {
        if (otaCancelled) { Update.abort(); return false; }  // ⑧ 支持取消
        size_t toRead = min(available, sizeof(buffer));
        int bytesRead = stream->readBytes(buffer, toRead);
        Update.write(buffer, bytesRead);  // ⑨ 写入 Flash
        totalRead += bytesRead;

        int progress = (totalRead * 100) / contentLength;
        if (progress % 5 == 0)  // ⑩ 每 5% 通过 BLE Notify 上报进度
            ota_notify_status(OTA_STATUS_INSTALLING, progress);
    }
    Update.end(true);  // ⑪ 完成写入，标记新分区为启动分区
}
⑤⑥HTTP → Flash 流式写入：不把整个固件下载到 RAM（放不下），而是 1KB 块边下载边写 Flash。ESP32 的 Update 库自动处理分区切换 — 写入非活跃的 App0/App1 分区。
⑩⑪进度上报 + 安全写入：手机通过 BLE Notify 实时看到升级进度。Update.end(true) 验证写入完整性，通过后设置 otadata 指向新分区。重启后运行新固件，失败则回滚。

void loop_app() {
    unsigned long now = millis();

    handleButton();    // ① 按钮处理（消抖 + 长短按判断）
    updateLED();       // ② LED 状态显示
    ota_loop();        // ③ OTA 处理（当前为空）

    // ④ 音频处理 — 最高优先级
    if (audioEnabled && mic_is_running()) {
        mic_process();   // 读取 I2S → 增益 → 回调
        opus_process();  // 环形缓冲区 → Opus 编码 → 回调
    }
    if (connected && audioSubscribed)
        processAudioTx();  // ⑤ BLE 发送编码音频

    // ⑥ 省电检查
    if (!connected && (now - lastActivity > IDLE_THRESHOLD_MS))
        enterPowerSave();
    else if (connected || photoDataUploading)
        if (powerSaveMode) exitPowerSave();

    // ⑦ 电池监测（20s 间隔）
    if (now - lastBatteryCheck >= BATTERY_TASK_INTERVAL_MS) {
        readBatteryLevel();
        updateBatteryService();
    }

    // 照片传输（限流：每轮最多 2 块，音频优先）...
    // ⑧ Light Sleep（无音频活跃时）
    if (!photoDataUploading && !audioSubscribed)
        enableLightSleep();

    // ⑨ 自适应延时
    if (photoDataUploading || audioSubscribed) delay(5);   // 活跃：5ms
    else delay(50);                                         // 空闲：50ms
}
④⑤音频优先：音频处理在照片传输之前。照片传输限流为每轮最多 2 块，且当音频缓冲区有数据时立即让步。这是协作式调度的核心 — 没有优先级抢占，靠代码中的优先级判断实现。
⑧⑨功耗自适应：无音频时进入 Light Sleep，活跃时 5ms 轮询，空闲时 50ms 轮询。整个主循环没有阻塞等待 — 所有操作都是非阻塞的，用时间戳判断是否执行。

可借鉴的设计模式

集中式配置

config.h 一个文件管理所有参数 — 引脚、BLE UUID、音频参数、功耗阈值。修改配置只改一个地方。STM32 项目可用同类的 board.h 或 conf.h。

回调链驱动数据流

Mic → onMicData → Opus ring buffer → opus_process → onOpusEncoded → TX ring buffer → processAudioTx → BLE Notify。每一层不知道上下层细节，只通过回调通信。

PSRAM 大容量缓冲

ESP32-S3 的 8MB PSRAM 用于音频环形缓冲区（16KB）和相机帧缓冲。内部 512KB RAM 留给栈和关键变量。STM32 没有 PSRAM，需要更精细的内存规划。

环形缓冲区

两级环形缓冲：PCM 环形缓冲（mic→opus）+ TX 环形缓冲（opus→BLE）。解耦生产者和消费者速率差异。嵌入式音频流的经典模式。

BLE 分帧传输协议

3 字节头（帧索引 + 子索引）+ payload。帧索引用于接收端排序和去重。照片传输用类似的帧索引机制。设计简洁实用。

协作式主循环调度

loop_app() 不用 RTOS 任务优先级，靠代码顺序 + 条件判断实现优先级：音频 > 照片 > 功耗检查。适合单线程 Arduino 框架。对比 FreeRTOS 的抢占式调度各有优劣。

从 omiGlass 到 Cortex-M：知识迁移路线

1

I2S PDM → STM32 SAI / nRF5340 PDM

ESP32 的 I2S PDM 模式等价于 STM32 的 SAI PDM 模式或 nRF5340 的硬件 PDM 外设。采样率配置、DMA 缓冲区设计、增益处理可直接迁移。

2

Opus → LC3（编解码器设计理念通用）

帧长、码率、复杂度的参数调优逻辑相同。Opus 用 16kHz/32kbps/20ms，LC3 用 48kHz/96kbps/10ms — 都是采样率/码率/延迟的三角权衡。

3

BLE Notify → BLE ISO（传输协议演进）

omiGlass 的 BLE Notify + 应用层协议是"手动挡"，nRF5340 Audio 的 ISO 同步通道是"自动挡"。理解手动挡有助于理解自动挡为什么更好。

4

ESP32 功耗管理 → STM32 Stop/Standby

CPU 降频 ≈ STM32 降低主频；Light Sleep ≈ STM32 Sleep；Deep Sleep ≈ STM32 Standby。策略相同：分级省电 + 唤醒源配置。

5

Arduino HAL → HAL 库 → 寄存器直操作（抽象层级对比）

omiGlass 用 Arduino HAL（i2s_driver_install 等），屏蔽了寄存器细节。STM32 用 HAL 库（HAL_I2S_Receive_DMA），或直接操作寄存器。从 Arduino 入门 → 理解 HAL → 直操作寄存器，是学习路径。
