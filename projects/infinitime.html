SystemTask::Work() — 主循环

SystemTask 是 InfiniTime 的核心。它创建其他任务，然后进入消息队列循环，根据收到的消息分发给各子系统。

📁 src/SystemTask.cpp 主任务入口 
void SystemTask::Work() {                    ①
    // 1. 初始化硬件驱动
    spi.Init();                               ②
    display.Init();                           // ST7789 显示屏初始化
    touchPanel.Init();                        // 触摸面板

    // 2. 启动 BLE 协议栈
    bleManager.Init();                        ③

    // 3. 创建显示任务
    displayApp = new DisplayApp();            ④
    displayApp->Start();

    // 4. 主消息循环
    while (true) {
        // 等待消息 (可被中断唤醒)
        if (xQueueReceive(systemQueue, &msg,  ⑤
                          pdMS_TO_TICKS(100)))
        {
            switch (msg.command) {
                case Messages::GoToSleep:     ⑥
                    displayApp->PushMessage(
                        DisplayApp::Messages::GoToSleep);
                    spi.Sleep();
                    isSleeping = true;
                    break;

                case Messages::WakeUp:        ⑦
                    spi.Wakeup();
                    display->Wakeup();
                    isSleeping = false;
                    break;

                case Messages::OnTouchEvent:  ⑧
                    displayApp->PushMessage(
                        DisplayApp::Messages::TouchEvent);
                    break;
            }
        }

        // 5. 空闲时进入低功耗
        if (isSleeping) {
            vTaskDelay(pdMS_TO_TICKS(1000));  ⑨
        }
    }
}
①这是 FreeRTOS 任务的入口函数。由 xTaskCreate() 调度执行。任务函数通常不返回（无限循环）
②nRF52 的 SPI 外设（SPIM）同时用于驱动 ST7789 显示屏和外部 SPI Flash。初始化包括配置引脚、时钟频率、模式（CPOL/CPHA）
③NimBLE 是 Apache BLE 协议栈的轻量级实现。初始化包括设置设备名、广播参数、GATT 服务（如心率服务、通知服务）
④DisplayApp 是独立任务，运行 LVGL 渲染循环。分离 UI 渲染和系统管理是嵌入式 GUI 的最佳实践——避免阻塞系统任务
⑤xQueueReceive 阻塞等待消息。100ms 超时意味着即使没有消息，每秒也会醒来 10 次执行空闲逻辑。这是消息驱动架构的核心
⑥收到 GoToSleep 时：先通知显示任务也进入睡眠，然后将 SPI 外设设为低功耗模式，设置 isSleeping 标志
⑦唤醒是睡眠的逆序：先恢复 SPI，再唤醒显示驱动。时序很重要——SPI 必须先就绪，否则显示命令会失败
⑧触摸事件由中断产生，通过 systemQueue 传到 SystemTask，再转发给 DisplayApp。这种"间接传递"模式解耦了中断和 UI
⑨睡眠期间降低任务频率，配合 FreeRTOS 的 tickless idle 模式减少功耗。实际效果：PineTime 的待机功耗可降到 1mA 以下

St7789 显示驱动 — SPI 实战

ST7789 是 240x240 TFT LCD 的控制芯片。通过 SPI 发送命令和数据来控制显示内容。

📁 src/drivers/St7789.cpp (简化) SPI 显示驱动 
void St7789::Init() {                        ①
    // 硬件复位: RESET 引脚拉低再拉高
    nrf_gpio_pin_clear(pinReset);
    nrf_delay_ms(10);
    nrf_gpio_pin_set(pinReset);
    nrf_delay_ms(120);

    // 软件复位命令
    WriteCommand(0x01);                       ②
    nrf_delay_ms(120);

    // 退出睡眠模式
    WriteCommand(0x11);                       ③
    nrf_delay_ms(120);

    // 设置像素格式: 16-bit RGB565
    WriteCommand(0x3A);
    WriteData(0x05);                          ④

    // 正常显示模式
    WriteCommand(0x13);                       // NORON
    nrf_delay_ms(10);

    // 开启显示
    WriteCommand(0x29);                       ⑤
}

void St7789::WriteCommand(uint8_t cmd) {
    nrf_gpio_pin_clear(pinDataCommand);       ⑥
    spi.Write(&cmd, 1);                      ⑦
}

void St7489::WriteData(uint8_t data) {
    nrf_gpio_pin_set(pinDataCommand);         ⑧
    spi.Write(&data, 1);
}

void St7789::SetAddrWindow(uint16_t x0, uint16_t y0,
                           uint16_t x1, uint16_t y1) {
    WriteCommand(0x2A);  // CASET: 列地址设置
    WriteData16(x0); WriteData16(x1);
    WriteCommand(0x2B);  // RASET: 行地址设置
    WriteData16(y0); WriteData16(y1);
    WriteCommand(0x2C);  // RAMWR: 写入显存    ⑨
}

void St7789::Sleep() {
    WriteCommand(0x28);  // DISPOFF: 关闭显示
    WriteCommand(0x10);  // SLPIN: 进入睡眠    ⑩
    nrf_delay_ms(120);
}
①显示芯片的初始化序列必须严格按照数据手册规定的顺序执行。复位 → 延时 → 命令配置 → 延时。跳过任何延时可能导致芯片行为异常
②SWRESET（0x01）是软件复位命令。某些场景下可以替代硬件复位引脚操作
③SLPOUT（0x11）让显示控制器退出低功耗状态。需要 120ms 等待内部 DC-DC 转换器稳定
④RGB565 格式：每个像素 16 位（R=5bit, G=6bit, B=5bit）。嵌入式 LCD 的标准格式，比 24-bit RGB 节省 33% 显存
⑤DISPON（0x29）开启显示输出。至此初始化完成，屏幕开始显示（虽然还是黑屏）
⑥DC（Data/Command）引脚是 SPI 显示的标准控制线。DC=LOW 告诉 ST7789 接下来发送的是命令字节
⑦SPI 发送命令字节。对于 nRF52，实际调用的是 SPIM 外设的 DMA 传输（NRF_SPIM_Task_START）
⑧DC=HIGH 切换到数据模式。ST7789 将接下来的字节理解为命令的参数
⑨RAMWR 命令后可以连续写入像素数据，无需重复命令。这就是 LVGL 渲染后刷新屏幕的核心机制
⑩进入睡眠模式可大幅降低显示芯片功耗。配合 SystemTask 的 GoToSleep 消息，实现手表的息屏省电

低功耗管理 — Tickless Idle

手表的续航核心是低功耗。InfiniTime 利用 FreeRTOS 的 tickless idle 机制，在空闲时停止 SysTick，让 CPU 进入深度睡眠。

📁 src/SystemTask.cpp (低功耗部分) 电源管理 
// FreeRTOS tickless idle 钩子
extern "C" void vApplicationIdleHook(void) { ①
    if (isSleeping) {
        // 计算可以睡眠的最大时间
        TickType_t xExpectedIdleTime =
            GetExpectedIdleTime();           ②

        // 停止 SysTick，配置 RTC 唤醒
        vPortSuppressTicksAndSleep(
            xExpectedIdleTime);              ③
    }
}

// 实际的 CPU 睡眠实现
void EnterLowPower(void) {
    // 1. 关闭不必要的外设时钟
    nrf_spim_disable(NRF_SPIM0);             ④

    // 2. 配置唤醒源 (GPIO 中断 或 RTC)
    sd_power_system_off_set(false);          ⑤

    // 3. 执行 WFI (Wait For Interrupt)
    __WFI();                                 ⑥

    // 4. 唤醒后恢复外设
    nrf_spim_enable(NRF_SPIM0);             ⑦
}
①FreeRTOS 空闲钩子在所有任务都阻塞时被调用。这是执行低功耗逻辑的最佳时机——没有任务需要运行
②计算距下一个定时器到期或任务唤醒的时间。这段时间内 SysTick 可以安全停止
③vPortSuppressTicksAndSleep 是 FreeRTOS tickless idle 的核心 API。它停止 SysTick，用低功耗定时器（RTC）替代，到期后恢复 SysTick 并补上漏掉的 tick
④睡眠前关闭外设时钟可以显著降低功耗。nRF52 的外设时钟独立于 CPU 时钟，可以单独开关
⑤nRF52 有多种低功耗模式：System OFF（最低功耗，需复位唤醒）和 System ON Idle（可被中断唤醒）。这里用后者
⑥__WFI() 是 Cortex-M 的 WFI 指令（Wait For Interrupt）。CPU 暂停执行，直到 NVIC 检测到中断请求。这是最低功耗的"等中断"方式
⑦唤醒后必须重新使能之前关闭的外设。顺序与睡眠时相反（后关的先开）

本节要点

  • 1. "星形任务"架构：SystemTask 作为中央协调器，通过 FreeRTOS 消息队列分发事件给子系统
  • 2. SPI 显示驱动使用 DC 引脚区分命令和数据模式，初始化序列必须严格按数据手册的顺序和延时
  • 3. tickless idle 机制：空闲时停止 SysTick，用 RTC 替代计时，配合 WFI 指令进入深度睡眠
  • 4. 睡眠/唤醒是对称操作：先关的后开。外设关闭顺序影响唤醒可靠性
  • 5. 消息队列解耦了中断处理和任务逻辑——中断只发消息，不做耗时操作（如 SPI 传输）
