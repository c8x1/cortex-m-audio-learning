Device Tree — 硬件声明式配置

Zephyr 最大的特色是 Device Tree。硬件配置不写在 C 代码里，而是声明在 .overlay 文件中。编译时，Zephyr 构建系统将 Device Tree 转换为 C 宏和结构体。

📁 boards/zswatch_nrf5340.overlay Device Tree 覆盖文件 
// SPI 总线配置 (连接显示屏)
&spi1 {                                    ①
    compatible = "nordic,nrf-spi";          ②
    status = "okay";                        ③
    cs-gpios = <&gpio0 17 GPIO_ACTIVE_LOW>;④

    gc9a01: gc9a01@0 {                      ⑤
        compatible = "galaxycore,gc9a01";   ⑥
        reg = <0>;                          // SPI 设备地址 (CS 索引)
        spi-max-frequency = <32000000>;     ⑦
        cmd-data-gpios = <&gpio0 14 0>;     ⑧
        reset-gpios = <&gpio0 15 GPIO_ACTIVE_LOW>;
        width = <240>;
        height = <240>;
        pixel-format = <PIXEL_FORMAT_RGB_565>;
    };
};

// I2C 总线配置 (连接传感器)
&i2c1 {
    status = "okay";
    pinctrl-0 = <&i2c1_default>;           ⑨

    bma456: bma456@68 {                     ⑩
        compatible = "bosch,bma456";
        reg = <0x68>;
        int-gpios = <&gpio1 5 GPIO_ACTIVE_LOW>;
    };
};
①&spi1 引用 SoC 级 Device Tree 中定义的 SPI1 节点，在此基础上添加板级配置。这是 Device Tree 的"叠加"（overlay）机制
②compatible 是 Device Tree 的核心属性，用于匹配驱动程序。构建系统根据此字符串找到对应的 Zephyr 驱动
③SoC 级 Device Tree 中大多数外设默认 status = "disabled"。板级 overlay 通过 status = "okay" 启用板子上实际存在的外设
④CS（片选）引脚：&gpio0 引用 GPIO 控制器，17 是引脚号，GPIO_ACTIVE_LOW 表示低电平有效
⑤gc9a01@0 定义一个 SPI 从设备。@0 是芯片选择索引（对应 cs-gpios 中的第一个引脚）
⑥驱动匹配：Zephyr 构建系统扫描所有 compatible 字符串，自动包含匹配的驱动源文件。不需要手动 #include
⑦告诉驱动 SPI 时钟上限。驱动会自动计算分频系数，不需要手动配置寄存器
⑧DC 引脚用于 SPI 显示的命令/数据切换。等价于 InfiniTime 中手动操作 nrf_gpio_pin_set/clear(pinDataCommand)
⑨nRF53 系列使用 pinctrl（引脚控制器）子系统管理引脚复用。引脚映射定义在单独的 pinctrl 节点中
⑩BMA456 加速度计挂在 I2C 总线上，7 位地址 0x68。中断引脚 GPIO1_5 用于 motion detect 唤醒

Kconfig — 功能开关

Kconfig 是 Zephyr 的编译配置系统（类似 Linux 内核的 menuconfig）。通过 prj.conf 文件启用或关闭功能模块。

📁 prj.conf 项目配置文件 
# RTOS 基础配置
CONFIG_MAIN_STACK_SIZE=2048                ①
CONFIG_SYSTEM_WORKQUEUE_PRIORITY=7

# 显示子系统
CONFIG_DISPLAY=y                           ②
CONFIG_SPI=y                               ③
CONFIG_LCD=y
CONFIG_LVGL=y                              ④
CONFIG_LV_Z_VDB_SIZE=25

# 传感器子系统
CONFIG_SENSOR=y                            ⑤
CONFIG_I2C=y
CONFIG_BMI270=y                            ⑥

# 蓝牙
CONFIG_BT=y                                ⑦
CONFIG_BT_PERIPHERAL=y
CONFIG_BT_DEVICE_NAME="ZSWatch"

# 电源管理
CONFIG_PM=y                                ⑧
CONFIG_PM_DEVICE=y
CONFIG_PM_POLICY_APP=y                     ⑨
①栈大小以字节为单位。2048 字节 = 2KB。Zephyr 的栈由内核动态分配（线程栈），不像裸机那样在链接脚本中静态分配
②=y 表示启用。对应 =n 表示禁用。显示子系统是 SPI 显示驱动的上层抽象
③Zephyr 的 SPI 驱动由 Kconfig 控制。启用后，构建系统自动包含 SPI 核心代码 + 与 Device Tree compatible 匹配的具体驱动
④Zephyr 内置 LVGL 集成（CONFIG_LVGL），不需要手动移植。LV_Z_VDB_SIZE 控制显示缓冲区占屏幕的百分比
⑤Zephyr 的 sensor API 提供统一的传感器访问接口。无论底层是 I2C 还是 SPI，上层代码都用同一套 API
⑥启用特定传感器驱动。构建系统只编译被启用的驱动，不会把所有驱动都编进去——这比 STM32 HAL 的"全包含"方式节省大量 Flash
⑦Zephyr 内置 BLE 协议栈（基于 Apache NimBLE），不需要像 InfiniTime 那样手动集成。CONFIG_BT_PERIPHERAL 启用外设角色
⑧Zephyr 的电源管理（PM）子系统提供设备级和系统级功耗控制。与 FreeRTOS 需要手动实现 tickless 不同，Zephyr PM 自动处理
⑨应用自定义电源策略：应用可以注册回调决定何时进入低功耗状态，比固定策略更灵活

传感器集成 — Zephyr Sensor API

Zephyr 提供了统一的传感器抽象层。不管底层是 I2C 还是 SPI，应用代码都用同一套 API。以下展示 IMU 传感器的使用模式。

📁 src/sensor_manager.c (简化) 传感器采集与手势检测 
// 获取 Device Tree 中定义的传感器设备
const struct device *imu =                  ①
    DEVICE_DT_GET(DT_NODELABEL(bma456));

void sensor_init(void) {
    if (!device_is_ready(imu)) {            ②
        printk("IMU not ready!\n");
        return;
    }

    // 配置加速度计: ±4g, 100Hz 采样
    struct sensor_value odr = {
        .val1 = 100,  // Hz
        .val2 = 0,
    };
    sensor_attr_set(imu, SENSOR_CHAN_ACCEL_XYZ,
        SENSOR_ATTR_SAMPLING_FREQUENCY, &odr);  ③

    // 注册数据就绪触发器
    struct sensor_trigger trig = {
        .type = SENSOR_TRIG_DATA_READY,
        .chan = SENSOR_CHAN_ACCEL_XYZ,
    };
    sensor_trigger_set(imu, &trig,         ④
                       on_imu_data);
}

void on_imu_data(const struct device *dev,
                 const struct sensor_trigger *trig) {
    struct sensor_value accel[3];

    // 从传感器 FIFO 读取最新样本
    sensor_sample_fetch(dev);               ⑤

    // 获取 X/Y/Z 三轴加速度
    sensor_channel_get(dev,                 ⑥
        SENSOR_CHAN_ACCEL_XYZ, accel);

    // 检测抬腕手势 (简单的阈值判断)
    float mag = sqrt(accel[0].val1 * accel[0].val1 +
                     accel[1].val1 * accel[1].val1 +
                     accel[2].val1 * accel[2].val1);
    if (mag > WAKE_THRESHOLD) {
        pm_device_state_set(display,        ⑦
            PM_DEVICE_STATE_ACTIVE, NULL, NULL);
    }
}
①DEVICE_DT_GET 宏在编译时展开为设备结构体的引用。不需要运行时查找——设备地址在编译时就确定了。这比 FreeRTOS 的手动初始化安全得多
②驱动初始化可能失败（I2C 通信错误、传感器未焊接等）。device_is_ready() 检查驱动的初始化标志，确保设备可用
③sensor_attr_set 是通用的传感器配置 API。不同驱动内部会翻译为具体的寄存器写入（如 BMA456 的 0x40 寄存器）
④触发器模式：传感器数据就绪时，硬件产生中断，Zephyr 在中断上下文中调用此回调。等价于裸机编程中的 EXTI 中断处理
⑤sensor_sample_fetch 从传感器的 FIFO 或数据寄存器读取原始值。对于 BMA456，实际执行 I2C 读操作
⑥sensor_channel_get 将原始 ADC 值转换为物理单位（m/s²）。sensor_value 结构体用 val1 + val2*10^-6 表示浮点数，避免使用 float
⑦检测到抬腕手势后，通过 Zephyr PM API 将显示设备从低功耗状态切换到活跃状态。PM 子系统自动处理时序和电源轨切换

选择建议

如果目标是理解底层原理（寄存器、中断、DMA），先读 InfiniTime。如果目标是快速开发产品，Zephyr 的 Device Tree + Kconfig 体系能大幅减少"从零配置外设"的时间。后续的 nRF5340 Audio 项目也使用 Zephyr，届时你会更深入体验这套体系。

本节要点

  • 1. Device Tree 是硬件的"声明式配置"：引脚映射、总线参数、设备地址都在 .overlay 文件中定义，不写在 C 代码里
  • 2. Kconfig 控制编译选项：只有 CONFIG_X=y 的模块才会被编译进固件，自动裁剪未使用的代码
  • 3. Zephyr Sensor API 提供统一接口：sensor_sample_fetch + sensor_channel_get，屏蔽底层差异
  • 4. Zephyr PM 子系统自动管理设备电源状态，无需手动实现 tickless idle
  • 5. Zephyr 的学习曲线更陡，但在复杂项目中（如双核 BLE Audio）能显著降低工程复杂度
