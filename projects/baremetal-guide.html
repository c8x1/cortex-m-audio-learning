Cortex-M 启动序列

当你按下复位按钮或上电时，Cortex-M 处理器执行一段固定的硬件流程，然后跳转到软件世界。这段流程发生在 main() 之前，是所有嵌入式程序的基础。

核心理解

Cortex-M 的启动是硬件固定流程，不像 x86 需要 BIOS。CPU 上电后直接从向量表地址 0x00000000 读取初始栈指针和复位处理函数地址。

Reset_Handler 流程（伪代码）

以下是每个 Cortex-M 芯片上电后第一段执行代码的完整流程。虽然不同厂商的实现细节不同，但核心步骤完全一致：

📁 startup.c (伪代码 — 通用 Cortex-M) 启动流程总览 
// Reset_Handler — 上电后第一个执行的软件函数
void Reset_Handler(void) {                ①

    // 步骤 1: 初始化栈指针 (通常已在硬件自动完成)
    // SP 已由硬件从向量表第 0 项加载

    // 步骤 2: 复制 .data 段从 Flash 到 RAM
    uint32_t *src = &_sidata;             ②
    uint32_t *dst = &_sdata;              ③
    while (dst < &_edata) {               ④
        *dst++ = *src++;
    }

    // 步骤 3: 清零 .bss 段
    dst = &_sbss;                         ⑤
    while (dst < &_ebss) {
        *dst++ = 0;                       ⑥
    }

    // 步骤 4: 系统初始化
    SystemInit();                         ⑦

    // 步骤 5: C++ 运行时初始化
    __libc_init_array();                  ⑧

    // 步骤 6: 进入 main
    main();                               ⑨

    // 如果 main 返回，进入死循环
    while (1);
}
①硬件从向量表的第 2 项（偏移 0x04）读取此函数的地址，然后跳转过来执行。这就是 main() 之前的世界
②_sidata 是链接脚本定义的符号，指向 Flash 中 .data 段的加载地址（LMA）。.data 段的初始值存储在 Flash 中
③_sdata 是 .data 段在 RAM 中的虚拟地址（VMA）。程序运行时，全局变量在这个地址被访问
④_edata 标记 .data 段末尾。将所有已初始化全局变量的值从 Flash 复制到 RAM，这样程序才能正确读写它们
⑤_sbss 标记 .bss 段的起始。BSS = Block Started by Symbol，存放未初始化或初始化为零的全局/静态变量
⑥C 语言标准规定：未初始化的全局变量必须为零。如果跳过此步，变量值将是随机数（RAM 上电状态不定）
⑦通常由 CMSIS 提供的函数：配置系统时钟（如 HSI/HSE/PLL）、使能 FPU（Cortex-M4F/M7）、设置中断优先级分组
⑧遍历 .init_array 段，调用所有 C++ 全局对象的构造函数。如果不用 C++，可省略此步
⑨嵌入式程序的 main() 通常不返回。如果返回了，进入下方的死循环（硬件异常保护的最后一道防线）

向量表定义

向量表是 Cortex-M 中断系统的核心数据结构。它位于 Flash 的起始位置，告诉 CPU 每个中断/异常的处理函数在哪里。

📁 vector_table.c 最小向量表 
// 向量表 — 必须放在 Flash 起始位置 (0x00000000 或 0x08000000)
// 每个项是一个函数指针（4 字节），按固定顺序排列
__attribute__((section(".isr_vector")))     ①
const void (*g_pfnVectors[])(void) = {

    // 前 16 项是 Cortex-M 核心异常 (所有芯片相同)
    (void *)&_estack,                      ②
    Reset_Handler,                         ③
    NMI_Handler,                           // 偏移 0x08: 不可屏蔽中断
    HardFault_Handler,                     // 偏移 0x0C: 硬件故障
    MemManage_Handler,                     // 偏移 0x10: 内存管理错误 (MPU)
    BusFault_Handler,                      // 偏移 0x14: 总线错误
    UsageFault_Handler,                    // 偏移 0x18: 用法错误 (除零等)
    0, 0, 0, 0,                            // 偏移 0x1C-0x28: 保留
    SVC_Handler,                           // 偏移 0x2C: 系统服务调用
    DebugMon_Handler,                      // 偏移 0x30: 调试监控
    0,                                     // 偏移 0x34: 保留
    PendSV_Handler,                        ④
    SysTick_Handler,                       ⑤

    // 之后是芯片特定的外部中断 (数量由厂商决定)
    // STM32F4: WWDG_IRQ, PVD_IRQ, TAMPER_IRQ, ...
    // 数量可从几十到上百个
};
①__attribute__((section(".isr_vector"))) 告诉链接器将此数组放在特定段。链接脚本会将此段放在 Flash 的最起始位置
②第一项不是函数指针，而是栈顶地址。CPU 复位时自动将此值加载到 SP（栈指针）寄存器。_estack 由链接脚本定义，通常是 RAM 的末尾地址
③第二项是 Reset_Handler 的地址。CPU 复位时自动加载到 PC（程序计数器），开始执行启动代码
④PendSV（可挂起系统调用）是 RTOS 实现任务切换的关键中断。它的优先级设为最低，确保在其他中断完成后才执行上下文切换
⑤SysTick 是 Cortex-M 内核自带的 24 位倒计时定时器，FreeRTOS/Zephyr 用它产生周期性时钟节拍（通常 1ms）

链接脚本 (.ld)

链接脚本定义了内存布局：哪些地址是 Flash，哪些是 RAM，各个段怎么摆放。它还定义了启动代码需要的那些符号（_sdata, _edata 等）。

📁 STM32F4.ld (简化版链接脚本) 内存布局定义 
/* 定义芯片的物理内存区域 */
MEMORY                                  ①
{
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 1024K   ②
    RAM  (rwx)  : ORIGIN = 0x20000000, LENGTH =  128K   ③
}

/* 定义各个段的摆放规则 */
SECTIONS                                ④
{
    /* 向量表必须放在 Flash 最前面 */
    .isr_vector :
    {
        . = ALIGN(4);
        KEEP(*(.isr_vector))            ⑤
        . = ALIGN(4);
    } > FLASH

    /* 代码段 */
    .text :
    {
        *(.text)                        /* 所有 .text 段 */
        *(.text*)                       /* .text.* 子段 */
        *(.rodata)                      /* 只读常量数据 */
        *(.rodata*)
        _etext = .;                     /* 代码段结束标记 */
    } > FLASH

    /* 已初始化数据 — 加载到 Flash，运行时复制到 RAM */
    .data :
    {
        _sdata = .;                     ⑥
        *(.data)
        *(.data*)
        _edata = .;                     ⑦
    } > RAM AT > FLASH                  ⑧

    /* 未初始化数据 — 只在 RAM 中，不需要 Flash 空间 */
    .bss :
    {
        _sbss = .;                      ⑨
        *(.bss)
        *(.bss*)
        *(COMMON)
        _ebss = .;                      ⑩
    } > RAM

    /* 栈顶 — RAM 末尾 */
    _estack = ORIGIN(RAM) + LENGTH(RAM);⑪
}
①MEMORY 命令告诉链接器目标芯片有哪些物理内存区域、它们的起始地址和大小
②STM32F4 的 Flash 映射到 0x08000000。属性 rx 表示只读且可执行。向量表和代码都存在这里
③Cortex-M 的 RAM 固定从 0x20000000 开始（由系统总线映射决定）。属性 rwx 表示可读写执行
④SECTIONS 命令定义输入段如何合并为输出段，以及每个输出段放在哪个内存区域
⑤KEEP() 告诉链接器即使没有代码引用此段也不要删除。向量表可能没有被代码直接引用，但硬件需要它
⑥_sdata 是一个链接器符号（不是变量），代表 .data 段在 RAM 中的起始地址。启动代码用它作为复制目标
⑦_edata 标记 .data 段末尾。启动代码通过 _edata - _sdata 知道要复制多少字节
⑧> RAM AT > FLASH 是关键：VMA（虚拟地址）在 RAM 中（运行时地址），LMA（加载地址）在 Flash 中。启动代码负责把数据从 LMA 搬到 VMA
⑨_sbss 标记 .bss 段起始。启动代码从这里开始填零
⑩_ebss 标记 .bss 段末尾。注意 .bss 不需要 AT > FLASH，因为它的初始值就是零，不占 Flash 空间
⑪_estack 指向 RAM 末尾。Cortex-M 栈是向下生长的（满递减），所以栈顶就是 RAM 的最高地址。向量表第一项引用此值

本节要点

  • 1. Cortex-M 上电后硬件自动从地址 0x00000000 读取栈指针，从 0x00000004 读取 Reset_Handler 地址
  • 2. Reset_Handler 负责将 .data 段从 Flash 复制到 RAM，并将 .bss 段清零
  • 3. 向量表是函数指针数组，前 16 项是核心异常，之后是芯片外部中断
  • 4. 链接脚本定义内存区域（MEMORY）和段布局（SECTIONS），并通过 _sdata/_edata 等符号与启动代码协作
  • 5. .data 的 VMA 在 RAM、LMA 在 Flash，需要运行时复制；.bss 只在 RAM，需要清零
