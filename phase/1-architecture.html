Cortex-M 架构基础

理解 Cortex-M 处理器的编程模型、中断系统、启动流程和内存布局。这是所有后续学习的基础。

1.1 Cortex-M 家族与选型

Cortex-M 系列从低功耗的 M0 到高性能的 M55，覆盖不同应用场景。作为音频开发者，你最可能接触 M4F、M7 和 M33。

内核 流水线 FPU DSP 典型芯片 音频适用
M0/M0+3-stage无无STM32G0, nRF51不适合
M33-stage无无STM32F1, LM3S勉强
M4F3-stageSP 单精度SIMDSTM32F4, nRF52适合入门
M76-stageSP/DPSIMDSTM32H7, i.MX RT高性能首选
M333-stageSPSIMDnRF53, STM32L5低功耗+安全
M55—SP/DPHeliumi.MX RT700AI+音频

音频选型关键：FPU 是必须的 — 音频 DSP 大量浮点运算。M4F 的单精度 FPU 已经够用；M7 的双精度 FPU 和 6 级流水线提供更高性能余量，适合复杂算法（如多通道 ANC）。

人机协作

你选架构，AI 写配置

选 M4F 还是 M33、要不要 FPU、RAM 需要多大——这些架构决策只有你能做。选定之后，Device Tree / Kconfig / CubeMX 配置可以让 AI 生成。

1.2 编程模型：寄存器与模式

Cortex-M 只有两种特权模式：Handler 模式（中断服务函数中）和 Thread 模式（普通代码）。每种模式有 MSP（主栈指针）和 PSP（进程栈指针）。

通用寄存器 r0-r12，其中 r13 是栈指针(SP)，r14 是链接寄存器(LR，存返回地址)，r15 是程序计数器(PC)。特殊寄存器：

特殊寄存器一览 
// 程序状态寄存器 (xPSR)
// APSR: 条件标志 (N, Z, C, V, Q)
// IPSR: 当前中断号 (异常编号)
// EPSR: 执行状态 (Thumb 位, IT 块)

// PRIMASK — 中断屏蔽寄存器
__disable_irq();    // PRIMASK = 1, 关闭所有可屏蔽中断
__enable_irq();     // PRIMASK = 0, 恢复中断

// BASEPRI — 优先级屏蔽
__set_BASEPRI(5 << 4);  // 屏蔽优先级 ≥ 5 的中断 (M4 优先级 0最高)
                         // 注意：左移4位因为优先级只用高4位

// CONTROL — 控制寄存器
// bit[0]: 0=特权 Thread, 1=非特权 Thread
// bit[1]: 0=使用 MSP, 1=使用 PSP
// bit[2]: 0=FPU 不 lazy stacking, 1=启用 lazy stacking

实践建议：裸机程序只用 MSP 和特权模式就够了。当开始学 FreeRTOS 时，RTOS 会切到 PSP 给每个任务独立栈空间。PRIMASK 和 BASEPRI 在音频中断保护中非常重要。

可淡化

寄存器位操作 AI 随手写

MODER &= ~(3U << (4*2)) 这类位操作模式是标准样板。理解原理即可，具体写法交给 AI。你的时间花在理解"为什么这样配置"上。

1.3 NVIC 嵌套向量中断控制器

NVIC 是 Cortex-M 的核心组件。与 OS 框架中的中断不同，这里是硬件级别的中断嵌套和优先级管理。

优先级机制

  • • 数值越小优先级越高（0 是最高）
  • • 支持 抢占（Preemption）：高优先级打断低优先级
  • • 支持 尾链（Tail-chain）：连续中断零延迟切换
  • • STM32F4 用 4 位优先级 → 16 级（0-15）

音频中的 NVIC 应用

  • • DMA 半传输/全传输中断 → 音频回调
  • • SAI/I2S 错误中断 → underrun/overrun
  • • 定时器中断 → 精确采样率时钟
  • • 用 BASEPRI 保护音频处理临界区
nvic_audio_config.c 配置音频相关中断优先级 
void audio_interrupt_setup(void) {
    // ① DMA 中断优先级设为 5 — 音频数据搬运
    //    这是最高优先级，不能被其他中断打断
    HAL_NVIC_SetPriority(DMA1_Stream1_IRQn, 5, 0);
    HAL_NVIC_EnableIRQ(DMA1_Stream1_IRQn);

    // ② SAI (I2S) 中断优先级 6 — 略低于 DMA
    //    用于检测 underrun/overrun 错误
    HAL_NVIC_SetPriority(SAI1_IRQn, 6, 0);
    HAL_NVIC_EnableIRQ(SAI1_IRQn);

    // ③ SysTick 优先级最低（15）— RTOS 时间片
    //    确保不会干扰音频中断时序
    HAL_NVIC_SetPriority(SysTick_IRQn, 15, 0);
}

// DMA 半传输完成中断 — 音频回调入口
void DMA1_Stream1_IRQHandler(void) {
    // 这里直接处理前半缓冲区的音频数据
    // 典型延迟 = buffer_size / 2 / sample_rate
    // 例：256 samples / 48000Hz ≈ 5.3ms
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    if (DMA_GetFlagStatus(DMA1_Stream1, DMA_FLAG_HTIF1)) {
        DMA_ClearFlag(DMA1_Stream1, DMA_FLAG_HTIF1);
        // 通知音频处理任务
        xTaskNotifyFromISR(audioTaskHandle, 0, eIncrement,
                          &xHigherPriorityTaskWoken);
    }

    // 如果唤醒了更高优先级任务，触发上下文切换
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
① DMA 中断是音频系统的核心回调。当 DMA 完成半缓冲区传输时触发，告诉 CPU "可以处理前半部分数据了"。
② SAI 错误中断用于检测 FIFO underrun（DAC 没及时拿到数据）或 overrun（ADC 数据没及时读走）。音频中出现这个就意味着有 "咔嗒" 噪声。
③ SysTick 必须优先级最低！否则 1ms 的系统时钟中断会打断音频 DMA 回调，造成延迟抖动。这是初学者最常犯的错误。
不可替代

优先级策略是工程判断

DMA 抢占 SysTick 的排序不是代码问题，是架构决策。AI 可以生成 NVIC_SetPriority() 调用，但它不知道你的 BLE 栈和音频 DMA 会冲突。只有你能判断系统中断优先级的全局合理性。

人机协作

你定策略 AI 写实现

你描述"DMA 优先级 5，SAI 优先级 6，SysTick 最低"，Claude 生成完整 NVIC 初始化。瓶颈是你能否做出正确的优先级规划。

1.4 从上电到 main()

MCU 上电后并不是直接执行 main()。理解这个过程对调试硬故障（HardFault）和编写可靠代码至关重要。

sequenceDiagram
    participant PWR as 上电
    participant ROM as Boot ROM
    participant VT as 向量表
    participant SR as startup.s
    participant M as main()
    PWR->>ROM: 1. CPU 从 0x00000004 取 Reset_Handler
    ROM->>VT: 2. 读取向量表 (Flash 起始处)
    VT->>VT: 3. 加载 SP = 栈顶地址
    VT->>SR: 4. 跳转 Reset_Handler 地址
    SR->>SR: 5. 复制 .data 段 (Flash→RAM)
    SR->>SR: 6. 清零 .bss 段
    SR->>SR: 7. 调用 SystemInit() — 配置时钟
    SR->>M: 8. 调用 main()
startup.c — 典型的启动代码 
// 向量表 — 放在 Flash 最前面，CPU 上电时读取
__attribute__((section(".isr_vector")))
const uint32_t vector_table[] = {
    (uint32_t)&_estack,         // 初始栈指针 (MSP)
    (uint32_t)Reset_Handler,    // Reset
    (uint32_t)NMI_Handler,      // NMI
    (uint32_t)HardFault_Handler,// HardFault ← 最常遇到的错误
    (uint32_t)MemManage_Handler,// 内存管理错误
    (uint32_t)BusFault_Handler, // 总线错误
    (uint32_t)UsageFault_Handler,// 用法错误
    0, 0, 0, 0,                 // 保留
    (uint32_t)SVC_Handler,      // SVCall (RTOS 上下文切换用)
    (uint32_t)DebugMon_Handler, // Debug Monitor
    0,                          // 保留
    (uint32_t)PendSV_Handler,   // PendSV (RTOS 任务切换用)
    (uint32_t)SysTick_Handler,  // SysTick
    // 外部中断从这里开始...
    (uint32_t)WWDG_IRQHandler,  // IRQ 0
    (uint32_t)PVD_IRQHandler,   // IRQ 1
    // ... 芯片手册定义的全部中断向量
};

void Reset_Handler(void) {
    // 1. 把 .data 从 Flash 复制到 RAM
    uint32_t *src = &_sidata;
    uint32_t *dst = &_sdata;
    while (dst < &_edata) *dst++ = *src++;

    // 2. 清零 .bss 段（未初始化的全局变量）
    dst = &_sbss;
    while (dst < &_ebss) *dst++ = 0;

    // 3. 系统初始化（配置时钟树等）
    SystemInit();

    // 4. C 库初始化（如果使用 newlib）
    __libc_init_array();

    // 5. 终于进入 main()
    main();

    // 不应该返回到这里
    while (1);
}
! HardFault 是最常见的错误 — 空指针解引用、栈溢出、访问未映射内存都会触发。学会用寄存器排查：SCB->HFSR、SCB->MMFAR、SCB->BFAR。
★ SVC 和 PendSV 现在不用管，当你学 FreeRTOS 时会发现：SVC 用于启动第一个任务，PendSV 用于任务间上下文切换（最低优先级确保安全切换）。

常见坑：栈空间不够用。如果全局变量太多（.bss 和 .data）或递归太深，栈指针会溢出到非法内存区域，触发 HardFault。音频缓冲区特别大（如 48kHz × 2ch × 2bytes × 256samples = 256KB），一定要放在静态变量区或使用 heap。

不可替代

HardFault 时你要知道 PC 怎么来的

凌晨 3 点面对 HardFault 地址 0x08001234，你需要知道去 .map 文件查函数、去反汇编查指令。启动流程、向量表、栈帧布局是调试的基石。AI 无法 attach SWD 探针帮你读 CFSR 寄存器。

1.5 Linker Script — 内存地图

Linker Script (.ld) 定义了程序的内存布局。它告诉链接器：代码放哪里、数据放哪里、栈有多大。这对音频开发特别重要，因为需要精确控制大块音频缓冲区放哪里。

STM32F407.ld — 简化的链接脚本 
/* MCU 内存定义 */
MEMORY {
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 512K  /* 代码 + 只读数据 */
    RAM (rwx)   : ORIGIN = 0x20000000, LENGTH = 128K  /* 数据 + 栈 + 堆 */
    CCMRAM(rwx) : ORIGIN = 0x10000000, LENGTH = 64K   /* 核心 coupled 内存 */
}

/* 符号定义 — startup.c 中引用这些 */
_estack = ORIGIN(RAM) + LENGTH(RAM);  /* 栈顶 = RAM 末尾，向低地址生长 */

SECTIONS {
    .isr_vector : { KEEP(*(.isr_vector)) } > FLASH     /* 向量表必须在最前面 */

    .text : {
        *(.text)           /* 代码 */
        *(.rodata)         /* 只读数据 (const) */
    } > FLASH

    .data : {
        _sdata = .;        /* .data 段起始 — startup.c 用 */
        *(.data)
        _edata = .;        /* .data 段结束 */
    } > RAM AT > FLASH     /* 运行在 RAM，加载在 Flash (startup 复制) */

    .bss : {
        _sbss = .;
        *(.bss)            /* 未初始化的全局变量 — startup 清零 */
        *(COMMON)
        _ebss = .;
    } > RAM

    /* DMA 缓冲区 — 放在特定 section */
    .audio_buffers (NOLOAD) : {
        *(.audio_dma_buf)  /* 用 __attribute__((section(".audio_dma_buf"))) */
    } > RAM
}
1 CCMRAM (Core Coupled Memory) 是 STM32F4 的 64KB 额外 RAM，直接连 CPU 不经过总线矩阵。把频繁访问的音频数据放这里可以减少 DMA 争用。
2 DMA 缓冲区对齐很重要。如果 DMA 和 CPU 同时访问同一个 RAM bank，会有 wait state。最佳实践是 DMA 缓冲区 32 字节对齐。
人机协作

你规划内存布局 AI 写 .ld 文件

你决定"音频缓冲放 SRAM，查找表放 CCMRAM，栈大小 4K"，AI 生成对应的 MEMORY 和 SECTIONS 定义。关键是你的布局规划能力。

不可替代

总线仲裁是系统级决策

CCMRAM vs SRAM 的选择影响 DMA 和 CPU 的竞争。这种芯片特定的内存架构知识是 AI 无法替代的——它不知道你的 DMA stream 和 CPU 同时访问哪个 bank。

1.6 总线矩阵与内存架构

理解总线矩阵对音频性能至关重要：CPU、DMA、外设通过总线矩阵访问内存。如果多条总线同时访问同一个 RAM 区域，会产生等待状态（wait state），影响实时性。

音频优化策略：让 CPU 用 CCMRAM（代码/查找表），DMA 用 SRAM（音频缓冲区）。这样两条总线并行工作，零等待。libDaisy 项目就用了这个策略 — 我们在源码走读中会看到。

不可替代

总线冲突是音频性能瓶颈

DMA 和 CPU 同时访问同一个 RAM bank 的竞争需要你手动规避。这不是代码问题而是硬件拓扑问题。让 CPU 用 CCMRAM、DMA 用 SRAM 的策略需要理解总线矩阵——这正是 libDaisy 等项目的核心优化。
