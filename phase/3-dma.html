DMA 数据引擎

DMA（直接内存访问）是音频系统的核心。它让数据在外设和内存之间自动搬运，CPU 只需处理已就绪的音频块。

3.1 为什么音频必须用 DMA？

48kHz 采样率意味着每秒 48000 次数据传输。如果用 CPU 中断方式，每 20.83μs 就要进一次中断 — CPU 几乎没时间做 DSP 处理。DMA 可以完全独立地搬运数据。

中断方式（不推荐）

每 20.83μs 进一次中断

中断开销 ~2μs (入栈/出栈)

CPU 占用率 ~10% 仅为搬运数据

没有时间做复杂 DSP

DMA 方式（推荐）

每 5.3ms（256 samples）进一次中断

中断开销极小（只设标志/发通知）

CPU 占用率 ~0%（DMA 搬运）

99% CPU 时间留给 DSP

类比：DMA 就像雇了一个快递员帮你搬箱子。你（CPU）只需要在箱子上贴好标签（配置 DMA），告诉快递员从哪搬到哪，然后你就可以专心做 DSP 了。快递员搬完一批会通知你。

不可替代

CPU 占用率对比是架构思维

48kHz 16bit 立体声 DMA 传输只占 CPU 0.5%，手动搬运会占 30%+——这组数据决定了系统可行性。在开始写代码之前，你必须能用这种定量思维评估方案。AI 可以生成 DMA 代码，但无法帮你做"这个方案行不行"的判断。

3.2 Ping-Pong 双缓冲 — 音频核心模式

这是音频 DMA 最重要的概念。CPU 和 DMA 交替使用两块缓冲区，实现无间断音频流。

sequenceDiagram
    participant DMA as DMA 控制器
    participant BUF as 音频缓冲区
    participant CPU as CPU/DSP

    Note over BUF: [......Ping......][......Pong......]
    DMA->>BUF: 填充 Ping 前半
    DMA->>BUF: Ping 半满 → 中断通知 CPU
    CPU->>BUF: 处理 Ping 数据 (DSP)
    DMA->>BUF: 继续填充 Pong
    DMA->>BUF: Pong 半满 → 中断通知 CPU
    CPU->>BUF: 处理 Pong 数据 (DSP)
    DMA->>BUF: 循环回到 Ping 开头
    Note over DMA,CPU: 循环往复，永不间断
audio_dma.c — Ping-Pong 双缓冲实现 
// 音频缓冲区定义
#define AUDIO_BUF_SIZE  512   // 总缓冲区大小（sample 数）
#define HALF_BUF_SIZE   256   // 半缓冲区 = 一个 ping 或 pong 块

// 32 字节对齐 — 优化 DMA 传输和 cache line
ALIGN_32BYTES(int16_t audio_buffer[AUDIO_BUF_SIZE]);

// 音频回调函数指针
static audio_callback_t user_callback = NULL;

// ① 启动 DMA 双缓冲音频流
HAL_StatusTypeDef audio_start(int16_t *rx_buf, int16_t *tx_buf,
                              uint32_t size, audio_callback_t cb) {
    user_callback = cb;

    // 启动 SAI 接收（麦克风）+ 发送（扬声器）的 DMA
    // DMA Circular 模式：自动循环，不需要重启
    HAL_SAI_Receive_DMA(&hsai_rx, (uint8_t*)rx_buf, size);
    HAL_SAI_Transmit_DMA(&hsai_tx, (uint8_t*)tx_buf, size);

    return HAL_OK;
}

// ② DMA 半传输完成回调 — Ping 块就绪
void HAL_SAI_RxHalfCpltCallback(SAI_HandleTypeDef *hsai) {
    if (user_callback) {
        // 前半缓冲区（Ping）已填满麦克风数据
        // CPU 现在可以安全地读前半、写前半
        user_callback(audio_buffer,               // 输入：前半
                      audio_buffer,               // 输出：前半
                      HALF_BUF_SIZE);
    }
}

// ③ DMA 全传输完成回调 — Pong 块就绪
void HAL_SAI_RxCpltCallback(SAI_HandleTypeDef *hsai) {
    if (user_callback) {
        // 后半缓冲区（Pong）已填满
        // DMA 已经循环回到缓冲区开头，正在填充 Ping
        user_callback(&audio_buffer[HALF_BUF_SIZE], // 输入：后半
                      &audio_buffer[HALF_BUF_SIZE], // 输出：后半
                      HALF_BUF_SIZE);
    }
}

// ④ 用户音频回调 — 在这里做 DSP
void my_audio_process(int16_t *in, int16_t *out, uint32_t size) {
    // 在这里调用 DSP 函数
    // 例：简单的直通（bypass）
    for (uint32_t i = 0; i < size; i++) {
        out[i] = in[i];  // 直通
    }

    // 或者调用 CMSIS-DSP 函数：
    // arm_biquad_cascade_df1_f32(&eq_instance, float_in, float_out, size);
    // arm_fir_f32(&fir_instance, float_in, float_out, size);
}
① Circular 模式是关键。DMA 到达缓冲区末尾后自动回到开头继续传输，不需要软件干预。这就是 "环形缓冲区" 的硬件实现。
② HalfCplt = Ping 块就绪。DMA 正在填充 Pong，CPU 安全访问 Ping。时序保证：CPU 必须在 DMA 填完 Pong 之前处理完 Ping。
③ Cplt = Pong 块就绪。DMA 循环回 Ping 开头，CPU 安全访问 Pong。两个回调交替触发，形成 "乒乓" 节奏。
④ 实时约束：DSP 处理必须在半个缓冲区时间内完成。256 samples / 48kHz = 5.33ms。如果处理时间超过这个值，就会 underrun → 咔嗒声。
不可替代

双缓冲心智模型是音频核心

DMA 填 Ping 时 CPU 处理 Pong，然后交换。HalfCplt 回调在缓冲区前半就绪时触发——这个心智模型不理解，就无法调试 underrun。underrun 的表现是"音频有咔嗒声"，代码里没有任何明显错误。

3.3 缓冲区大小与延迟

缓冲区大小直接决定音频延迟。这是音频系统最重要的设计参数之一。

半缓冲区 48kHz 延迟 16kHz 延迟 DSP 时间 适用场景
32 samples0.67ms2.0ms很紧专业音频/ANC
64 samples1.33ms4.0ms适中低延迟音乐处理
128 samples2.67ms8.0ms充裕语音处理
256 samples5.33ms16.0ms宽裕BLE 音频/通用
512 samples10.67ms32.0ms很宽裕非实时处理

延迟计算公式：Latency = half_buffer_size / sample_rate。注意这是单程延迟 — 麦克风到处理完的总延迟要加上算法延迟（滤波器群延迟等）。ANC 通常要求 < 2ms 总延迟，BLE LC3 编码要求约 7.5-20ms。

不可替代

Buffer size ↔ Latency 是系统级决策

ANC 需要总延迟 <2ms，BLE LC3 容忍 7.5-20ms。32 samples = 0.67ms，256 samples = 5.33ms——选错 buffer size 无法通过调代码修复，它是架构决策，级联影响 DSP 时间预算、功耗、感知质量。

人机协作

你定延迟目标 AI 生成 DMA 配置

"256 samples double buffer, circular DMA, half-complete interrupt"——描述清楚后 AI 生成 HAL_DMA_Init 和回调注册。你的价值在于选择正确的 buffer size。

3.4 DMA Stream 和 Channel 路由

STM32F4 有两个 DMA 控制器，每个有 8 个 Stream。每个 Stream 只能映射到特定的外设请求（Channel）。音频通常需要多个 Stream 同时工作。

常见坑：Stream 冲突。同一个 Stream 不能同时用于两个请求。如果 SPI 和 I2S 都想用 DMA1 Stream1，只有一个能赢。解决方法：查看芯片 Reference Manual 的 DMA request mapping 表，提前规划。

人机协作

你规划分配表 AI 生成配置

你列出"I2S 用 DMA1 Stream5 Channel0，SPI 用 DMA2 Stream3 Channel3"，AI 生成完整 DMA_Init 代码。但你必须查 Reference Manual 的 DMA request mapping 表来确保不冲突。

不可替代

芯片特定路由表需手动查阅

SPI3_TX 和 I2S2_RX 可能都映射到 DMA1 Stream5——AI 不知道你的具体引脚分配。Stream 冲突会导致一个外设静默工作。这种 bug 只有查阅 RM 的 DMA request 表才能发现。
