RTOS 实时操作系统

从裸机循环（super-loop）到 RTOS 多任务。对比 FreeRTOS 和 Zephyr，理解音频任务如何与其他任务协调。

4.1 Super-loop vs RTOS

对比：裸机 vs RTOS 架构 
// ===== 裸机 Super-loop =====
// 优点：简单、可预测、无 RTOS 开销
// 缺点：所有功能耦合在一起，无法独立调度

void main(void) {
    hardware_init();
    while (1) {
        audio_process();    // 必须最快完成！
        ble_handler();      // 可能错过音频截止时间
        ui_update();        // 低优先级但占 CPU
        sensor_read();      // 每次循环都读，浪费
    }
}

// ===== RTOS 多任务 =====
// 优点：优先级调度、独立开发、实时保证
// 缺点：开销（RAM/ROM/上下文切换时间）

void audio_task(void *pv) {       // 最高优先级
    while (1) {
        ulTaskNotifyTake(pdTRUE, portMAX_DELAY); // 等 DMA 中断通知
        dsp_process();            // 实时处理，保证不被打断
    }
}

void ble_task(void *pv) {         // 中等优先级
    while (1) {
        vTaskDelay(pdMS_TO_TICKS(10));
        ble_stack_process();      // BLE 协议栈处理
    }
}

void ui_task(void *pv) {          // 低优先级
    while (1) {
        xSemaphoreTake(ui_mutex, portMAX_DELAY);
        lvgl_update();            // LVGL UI 刷新
    }
}

void main(void) {
    hardware_init();
    xTaskCreate(audio_task, "audio", 1024, NULL, 5, NULL);  // 优先级 5（最高）
    xTaskCreate(ble_task,   "ble",   512,  NULL, 3, NULL);  // 优先级 3
    xTaskCreate(ui_task,    "ui",    768,  NULL, 1, NULL);  // 优先级 1（最低）
    vTaskStartScheduler();
}

音频任务永远是最高优先级。RTOS 的抢占式调度保证：当 DMA 半传输中断通过 TaskNotify 唤醒音频任务时，它会立即抢占 BLE 和 UI 任务。这比 super-loop 可靠得多。

不可替代

任务划分和优先级是架构决策

音频任务必须在最高优先级、永远不能阻塞——这是设计原则不是代码。AI 可以创建任务，但只有你知道为什么音频任务必须抢占一切。

freertos_audio.c — FreeRTOS 音频管线 
// FreeRTOS 音频管线 — 实际项目中使用的模式

static TaskHandle_t audio_task_handle;
static StreamBufferHandle_t mic_stream;  // 麦克风数据流

// ① DMA 中断 → 通知音频任务
void DMA1_Stream1_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    if (dma_half_transfer_flag) {
        // 发送麦克风数据到 StreamBuffer
        // StreamBuffer 是 ISR-safe 的，适合音频数据传递
        xStreamBufferSendFromISR(mic_stream,
            &audio_buffer[0],           // Ping 数据
            HALF_BUF_SIZE * 2,          // 字节数
            &xHigherPriorityTaskWoken);
    }

    if (dma_full_transfer_flag) {
        xStreamBufferSendFromISR(mic_stream,
            &audio_buffer[HALF_BUF_SIZE], // Pong 数据
            HALF_BUF_SIZE * 2,
            &xHigherPriorityTaskWoken);
    }

    // ② 如果唤醒了高优先级任务，请求上下文切换
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

// ③ 音频处理任务
void audio_task(void *param) {
    int16_t rx_buf[256];

    while (1) {
        // 阻塞等待麦克风数据 — 超时 = 检测 underrun
        size_t received = xStreamBufferReceive(mic_stream,
            rx_buf, sizeof(rx_buf),
            pdMS_TO_TICKS(20));  // 超时 20ms

        if (received > 0) {
            // ④ 做实际的 DSP 处理
            arm_biquad_cascade_df1_f32(&eq_filter,
                float_in, float_out, received / 2);

            // ⑤ 发送处理后的数据到输出 StreamBuffer
            xStreamBufferSend(speaker_stream,
                float_out, received,
                pdMS_TO_TICKS(5));
        }
    }
}

// ⑥ 音频任务创建和 StreamBuffer 初始化
void audio_pipeline_init(void) {
    // StreamBuffer 大小必须 > 触发阈值，否则接收方不会被唤醒
    mic_stream = xStreamBufferCreate(2048,   // 总大小
                                      256);    // 触发阈值 (bytes)
    speaker_stream = xStreamBufferCreate(2048, 256);

    xTaskCreate(audio_task, "audio",
                1024,  // 栈大小 (words) = 4096 bytes
                NULL,
                configMAX_PRIORITIES - 1,  // 最高优先级
                &audio_task_handle);
}
① ISR 中使用 FromISR 变体。FreeRTOS 的 API 分为任务版本和 ISR 版本。ISR 版本不阻塞，通过 xHigherPriorityTaskWoken 参数决定是否需要上下文切换。
② portYIELD_FROM_ISR 是关键的上下文切换触发点。如果 ISR 唤醒了更高优先级的任务（音频任务），这里会直接切换到音频任务，而不是返回被中断的低优先级任务。
③ StreamBuffer 是 FreeRTOS 专门为音频/串口等连续数据流设计的。比 Queue 更高效（零拷贝语义，适合大数据块）。InfiniTime 项目大量使用它。
⑥ 触发阈值很重要：StreamBuffer 积累到 256 字节才会唤醒接收任务。太小会频繁唤醒（浪费 CPU），太大会增加延迟。
可淡化

FreeRTOS API 签名 AI 随手生成

xTaskCreate、xSemaphoreGive、vTaskDelay——这些标准 API 调用 AI 生成正确且高效。学习投资应放在理解何时用 Queue、何时用 Semaphore、何时用 StreamBuffer。

人机协作

你设计数据流 AI 实现 ISR-safe 代码

StreamBuffer 的零拷贝语义完美适合音频。你描述"ISR 中发 256 bytes，任务端等满 512 bytes"，AI 生成完整模式。

4.3 Zephyr RTOS — 现代替代方案

Zephyr 是 Linux 基金会主导的 RTOS，Nordic（nRF 系列）芯片的首选 RTOS。与 FreeRTOS 的最大区别：Device Tree、Kconfig、统一的设备驱动模型。

FreeRTOS

  • • 手动配置外设（CubeMX 生成的代码）
  • • C API，简单直接
  • • 轻量，RAM 开销小
  • • 适合：STM32、ESP32
  • • 代表项目：InfiniTime

Zephyr

  • • Device Tree 声明硬件（像 Linux）
  • • Kconfig 配置系统
  • • 统一驱动模型 + 电源管理
  • • 适合：nRF、i.MX RT
  • • 代表项目：ZSWatch、nRF5340 Audio
zephyr_audio.conf — Zephyr 音频配置 (Kconfig) 
# Zephyr 通过 Kconfig 配置系统功能
# 不需要手写初始化代码，配置即启用

# 启用 I2S 驱动
CONFIG_I2S=y

# 启用 Nordic nRFx 外设驱动
CONFIG_NRFX_I2S=y

# 音频 Codec 支持
CONFIG_AUDIO=y
CONFIG_AUDIO_CODEC=y

# 电源管理（对低功耗音频设备重要）
CONFIG_PM=y
CONFIG_PM_DEVICE=y

# BLE 音频支持
CONFIG_BT=y
CONFIG_BT_AUDIO=y
CONFIG_BT_AUDIO_UNICAST=y

# LC3 编解码器
CONFIG_LC3=y

# 调试：RTT 输出（替代 UART，更快）
CONFIG_DEBUG=y
CONFIG_LOG=y
CONFIG_LOG_BACKEND_RTT=y

选型建议：如果目标是 Nordic nRF53 系列（智能眼镜很可能用这个），选 Zephyr。如果目标是 STM32，FreeRTOS 生态更成熟。两个都要了解，因为面试和开源项目中两个都会遇到。

"nRF5340，I2S master，2 个 PDM 麦克风"——你用自然语言描述硬件，AI 生成 Device Tree overlay 和 Kconfig。

不可替代

Zephyr vs FreeRTOS 选型是架构决策

nRF = Zephyr，STM32 = FreeRTOS。选型基于芯片厂商支持、社区生态、项目规模——这是技术战略不是代码选择。

4.4 上下文切换与实时保证

RTOS 通过 PendSV 中断实现上下文切换。理解这个过程有助于估算音频任务的最坏响应时间。

sequenceDiagram
    participant IRQ as DMA 中断
    participant RTOS as FreeRTOS 调度器
    participant PEND as PendSV
    participant AUDIO as audio_task
    participant BLE as ble_task

    BLE->>BLE: 正在执行
    IRQ->>RTOS: DMA 半传输完成
    Note over IRQ: xTaskNotifyFromISR
唤醒 audio_task
    RTOS->>PEND: 触发 PendSV (最低优先级中断)
    PEND->>PEND: 保存 BLE 上下文 (R4-R11, PSP)
    PEND->>AUDIO: 恢复 audio_task 上下文
    Note over AUDIO: 总延迟 ≈ 中断延迟 (12周期)
+ ISR 时间 (~2μs)
+ 上下文切换 (~1μs)
≈ 3-5μs

实时保证：从 DMA 中断触发到音频任务开始执行，总延迟约 3-5μs（M4 @ 168MHz）。相比 5.33ms 的音频块处理时间，这个开销可以忽略。但前提是：音频任务的优先级最高，且不能在音频任务中调用阻塞 API。

不可替代

portYIELD_FROM_ISR 是正确性门槛

这个宏决定了 RTOS 是否立即切换到更高优先级的音频任务。用错变体音频时序会静默出错。这不是样板代码——它是实时系统的正确性门槛。
