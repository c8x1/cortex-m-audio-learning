6.1 低功耗设计

智能眼镜必须低功耗。理解 Cortex-M 的睡眠模式、Tickless Idle、外设时钟门控。

模式 CPU Flash 外设 唤醒时间 典型功耗
Sleep停止保持运行~1μs~2mA
Stop停止保持停止~5μs~20μA
Standby断电保持断电~50ms~2μA
Shutdown断电部分保持断电~100ms~0.1μA
low_power.c — FreeRTOS Tickless Idle 模式 
// FreeRTOS Tickless Idle：空闲时进入 Sleep 模式
// 在 FreeRTOSConfig.h 中启用：
// #define configUSE_TICKLESS_IDLE 1

// ① 空闲任务钩子 — 进入低功耗
void vApplicationIdleHook(void) {
    // 计算到下一个任务唤醒的时间
    TickType_t xExpectedIdleTime = prvGetExpectedIdleTime();

    if (xExpectedIdleTime > 2) {  // 至少 2 个 tick 才值得进入睡眠
        // ① 暂停 SysTick（它每 1ms 中断一次，太频繁）
        vPortSuppressTicksAndSleep(xExpectedIdleTime);
    }
}

// ② FreeRTOS 调用这个函数实现 Tickless
void vPortSuppressTicksAndSleep(TickType_t xExpectedIdleTime) {
    // 计算实际可以睡眠的 CPU 周期数
    uint32_t ulLowPowerTimeBeforeSleep;
    uint32_t ulLowPowerTimeAfterSleep;

    // 关中断
    __disable_irq();

    // 确认没有任务就绪
    if (eTaskConfirmSleepModeStatus() == eAbortSleep) {
        __enable_irq();
        return;
    }

    // 停止 SysTick，配置 RTC/LPTIM 做长周期唤醒
    ulLowPowerTimeBeforeSleep = ulPortGetTimerValue();
    SysTick->CTRL &= ~SysTick_CTRL_ENABLE_Msk;

    // ③ 进入 WFI (Wait For Interrupt) — CPU 停止
    // 只有中断能唤醒
    __WFI();

    // ④ 唤醒后恢复 SysTick，补偿流失的 tick
    ulLowPowerTimeAfterSleep = ulPortGetTimerValue();
    uint32_t elapsed_ticks = (ulLowPowerTimeAfterSleep - ulLowPowerTimeBeforeSleep)
                              / (configCPU_CLOCK_HZ / configTICK_RATE_HZ);
    vTaskStepTick(elapsed_ticks);

    SysTick->CTRL |= SysTick_CTRL_ENABLE_Msk;
    __enable_irq();
}
① Sleep 模式最适合音频设备。CPU 停止但 SRAM 和外设保持运行，DMA 仍在工作。唤醒只需 1μs，不影响音频时序。
②③ Tickless Idle的关键：暂停 1ms 的 SysTick 中断，用更长的唤醒周期替代。这样 CPU 在空闲时不会被 1ms 定时器频繁唤醒，省电。
④ Tick 补偿：唤醒后必须告诉 RTOS 过了多少时间。否则 FreeRTOS 的超时机制会出错（任务以为只过了 1ms，实际过了 100ms）。

6.2 调试技术

嵌入式调试和 OS 开发完全不同。没有 printf（太慢），没有 GDB 远程调试（大多数时候）。掌握 SWD、ITM、DWT。

SWD 调试接口

  • • 两线：SWDIO + SWCLK
  • • 可以实时读/写寄存器和内存
  • • 设断点、单步执行
  • • 工具：OpenOCD + GDB 或 SEGGER Ozone

ITM (Instrumentation Trace)

  • • 硬件级 printf — 不用 UART
  • • 通过 SWO 引脚输出（单线）
  • • 速度可达 2MHz，几乎零 CPU 开销
  • • 工具：SEGGER RTT / SWO Viewer
debug.c — DWT 周期计数器 + 硬件断点 
// ① DWT 周期计数器 — 精确测量代码执行时间
// DWT (Data Watchpoint and Trace) 是 Cortex-M 内置的调试外设

#define DWT_CYCCNT ((volatile uint32_t *)0xE0001004)

void dwt_init(void) {
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;  // 使能 DWT
    *DWT_CYCCNT = 0;                                   // 清零计数器
    DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;               // 启动计数
}

// 使用方法：测量 DSP 处理时间
uint32_t start = *DWT_CYCCNT;
arm_biquad_cascade_df1_f32(&eq, in, out, 256);
uint32_t cycles = *DWT_CYCCNT - start;
float us = (float)cycles / (SystemCoreClock / 1000000.0f);
// 例：168MHz CPU, 5000 cycles = 29.8μs

// ② HardFault 调试 — 最常见的崩溃
void HardFault_Handler(void) {
    // 读取故障寄存器
    uint32_t hfsr = SCB->HFSR;    // HardFault 状态
    uint32_t cfsr = SCB->CFSR;    // Configurable Fault 状态
    uint32_t mmfar = SCB->MMFAR;  // 内存管理错误地址
    uint32_t bfar = SCB->BFAR;    // 总线错误地址

    // 从汇编中获取 LR (返回地址) — 告诉你崩溃在哪
    __asm volatile (
        "TST LR, #4\n"           // 检查使用 MSP 还是 PSP
        "ITE EQ\n"
        "MRSEQ R0, MSP\n"        // MSP
        "MRSNE R0, PSP\n"        // PSP
        "B hard_fault_handler_c\n"
    );
}

void hard_fault_handler_c(uint32_t *stack_frame) {
    // stack_frame[0] = R0
    // stack_frame[1] = R1
    // ...
    // stack_frame[6] = PC  ← 崩溃发生的地址！
    // stack_frame[7] = PSR
    volatile uint32_t pc = stack_frame[6];

    // 用调试器查看 pc 的值，然后 addr2line 找到源码行
    __BKPT(0);  // 软件断点 — 调试器会停在这里
}
① DWT 周期计数器是性能分析的利器。比微秒计时器精确 168 倍（在 168MHz CPU 上）。用它测量每个 DSP 函数的执行时间，确保不超过音频块的时间预算。
② HardFault 排查三步：1) 读 HFSR/CFSR 确定错误类型；2) 读 MMFAR/BFAR 获取错误地址；3) 从栈帧读 PC 找到崩溃位置。这是面试常考的知识点。
不可替代

HardFault 诊断只能手动练

读 HFSR/CFSR/MMFAR/BFAR，从栈帧提取 PC，用 addr2line 定位源码——AI 无法 attach 你的 SWD 探针。三步诊断法必须练到肌肉记忆。

6.4 内存优化

音频设备 RAM 紧张（通常 128-512KB）。音频缓冲区就占掉很多，必须精打细算。

memory_budget.c — 音频系统的 RAM 预算 
// 典型 STM32F407 (192KB RAM) 的内存预算
// 128KB 主 SRAM + 64KB CCMRAM

// === 主 SRAM 分配 (128KB) ===
// DMA 音频缓冲区 (双缓冲)
ALIGN_32BYTES(int16_t audio_rx_buf[512]);  // 1KB (接收)
ALIGN_32BYTES(int16_t audio_tx_buf[512]);  // 1KB (发送)

// DSP 工作缓冲区
float32_t dsp_in[256];                     // 1KB
float32_t dsp_out[256];                    // 1KB
float32_t dsp_temp[256];                   // 1KB

// EQ 滤波器状态 (3 段 Biquad × 4 状态)
float32_t eq_state[12];                    // 48 bytes

// FIR 滤波器状态 + 系数
float32_t fir_state[128];                  // 512 bytes
float32_t fir_coeffs[64];                  // 256 bytes

// FreeRTOS 堆 (任务栈、队列、信号量)
// 需要约 20-30KB

// BLE 协议栈
// nRF52832 的 SoftDevice 需要 ~8KB
// Zephyr BLE 栈需要 ~16KB

// LVGL UI 缓冲区
// 240×240 × 2 bytes = ~115KB → 太大！
// 解决方案：部分刷新，用 SPI DMA 逐行传输

// === CCMRAM 分配 (64KB) — CPU 独占 ===
// 把音频查找表和常量放这里，避免和 DMA 争用总线
__attribute__((section(".ccmram")))
const float32_t sin_table[1024];            // 4KB 正弦查找表

__attribute__((section(".ccmram")))
float32_t cmsis_fft_buf[512];              // FFT 工作缓冲区

内存优化技巧：① 用 const 让编译器把只读数据放 Flash；② DMA 缓冲区放 SRAM，CPU 频繁访问的查找表放 CCMRAM；③ 音频缓冲区复用（录音和播放不用同一时间）；④ 使用 -Os 编译选项优化代码大小。
